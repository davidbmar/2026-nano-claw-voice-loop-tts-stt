Generic knowledge ingestion for voice AI

How the voice AI learns a website — for any website.

One pipeline, a small adapter per source. The same machine that lets the Space Channel voice answer about launches and UFO cases should answer about the JFK archive or an IBT news feed — by swapping a config, not forking the code. This is the map of every way we pull knowledge in, what's built today, and what's specced next.

THESIS

The source-connector pattern

Across four of our own repos the shape is identical: a stable five-stage pipeline with a thin, swappable adapter at the acquisition edge. It's the same idea as this codebase's provider registry — one interface, many backends. A new site is a new adapter + a config, never a new pipeline. Everything below hangs on that spine.

STAGE 01
Acquire
Per-source adapters pull raw data: crawl, feed capture, search & news APIs, document archives, files.
STAGE 02
Distill
Normalize into sourced records + a compact, cache-friendly digest. Full provenance attached.
STAGE 03
Dedup
Content-hash IDs drop repeats; overlap raises a corroboration score instead of adding noise.
STAGE 04
Index
Two axes: an always-on digest for identity, a vector/keyword index for scale and recall.
STAGE 05
Retrieve & Inject
Per turn: pick the relevant slice and inject it into the prompt — no tool, before the model speaks.

The voice constraint shapes stage 05: a knowledge tool pauses the loop for approval and wrecks the conversation, so retrieval must be a non-tool, pre-model step measured in tens of milliseconds — and it must always degrade to the digest rather than fail.

RULES

Cross-cutting principles

What makes it generic, safe, and voice-ready — independent of which site is plugged in.

◆ One interface, many adapters

The connector interface is the product; each site is an adapter + config. Space Channel, JFK, and IBT are three configs of one system.

◆ Provenance is a schema requirement

Every fact stores its source, type, and date, validated at ingest — not bolted on. This is what lets JFK-grade answers cite.

◆ Freshness by ledger, not by clock

Track last_collected per source and refresh the stalest first; a dirty-flag gate keeps expensive re-indexing rare.

◆ Voice retrieval is non-tool

Embedded store, model warmed at boot, selection under a millisecond, injected before the turn. A tool round-trip blows the budget.

◆ Always degrade, never fail

Retrieval down? Fall back to the digest. A stale answer beats a dead assistant — the same rule as the model fallback chain.

◆ Four uniform surfaces

Every corpus is reachable via CLI, REST, MCP, and Cron — so humans, agents, and schedules all use the same knowledge.

NEXT

Recommended path & prototype

Recommended for Space Channel

Hybrid pre-retrieval, digest as floor. Ship in latency-safe order, each step reversible behind a flag:

  1. Promote the BM25 spike to production selection — a non-tool step before providerManager.completeStream; inject the one selected detail file as a per-turn suffix. Core digest stays always-on; retrieval failure degrades to it.
  2. Broaden the eval beyond 8 clean questions — STT-mangled input, paraphrases, compound and follow-up questions, and no-match cases — before enabling by default.
  3. Add the vector axis (LanceDB + nomic-embed) for semantic recall where keywords miss, gated by the same latency budget.
  4. Adopt provenance records + citation tests so answers are sourced — the prerequisite for a JFK-grade corpus.
  5. Extract the adapter framework + ledger — at which point JFK and IBT become new configs, not new projects.

Prototype status: the file-level selection spike is built and reproduces (8/8, 0.014 ms). Next we design and test the broadened eval and the TypeScript hook — same swap-the-corpus test on a second site validates the "generic" claim.
